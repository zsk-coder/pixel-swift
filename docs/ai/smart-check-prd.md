> **版本**: v6.0 Workflow Copilot  
> **最后更新**: 2026-04-19  
> **推荐主路径**: `/workflow-copilot`  
> **文档说明**: 保留 `smart-check-prd.md` 文件名以兼容现有文档链路，但产品主线已从 `Smart Check / Image Audit` 升级为 `Workflow Copilot`。`Image Audit` 作为一个典型场景被纳入 Copilot 的任务模板中。

---

## 1. 产品定义

### 1.1 一句话定位

> **PixelSwift Workflow Copilot** 是一个面向图片处理场景的单轮规划式 AI Copilot。用户不再手动配置复杂参数，而是通过一句目标描述，让 AI 自动生成可执行的图片处理计划，并调用 PixelSwift 现有能力批量完成处理。

### 1.2 产品本质

本产品不是聊天机器人，也不是单一图片工具，而是一个：

- **自然语言任务输入层**
- **AI 规划层**
- **本地图片处理执行层**
- **结果导出层**

的完整工作流产品。

### 1.3 与现有产品的关系

- 现有的压缩、转换、尺寸调整、批量下载能力继续保留
- 这些能力不再只通过表单参数暴露，而是由 Copilot 编排调用
- `Image Audit`、`Amazon Smart Check`、`Shopify 商品图优化`、`博客配图压缩` 等都变成 Copilot 的任务模板或场景入口

---

## 2. 为什么做这个方向

### 2.1 当前问题

PixelSwift 已有强处理能力，但用户仍然需要自己决定：

- 这批图片应该压缩到什么程度
- 应该转成什么格式
- 是否需要统一尺寸
- 是否需要保留透明背景
- 应该如何命名与导出
- 对于 Shopify / Amazon / 博客场景，到底该组合哪些操作

这意味着当前产品仍然偏“工具箱”，而不是“决策型产品”。

### 2.2 新方向解决的问题

Workflow Copilot 解决的是：

> **用户知道目标，但不知道应该怎么配置处理参数时，系统如何自动做出合理决策。**

典型例子：

- “把这批图片处理成适合 Shopify 商品页的版本”
- “把博客配图压到 500KB 以内并转成 WebP”
- “为 Amazon 商品图准备一套可上架版本”
- “生成适合英文站的图片包和 SEO 元数据”

### 2.3 为什么这个方向 AI 是核心能力

这个方向里，AI 不再只是给出后置建议，而是负责：

- 理解用户意图
- 理解图片批次特征
- 结合模板与知识库做决策
- 生成结构化处理计划
- 对计划中的步骤给出原因和风险说明

没有 AI，用户仍然需要回到手工配置参数；有 AI，用户可以直接描述目标并获得完整执行计划。因此 AI 在该方向中是**前置决策入口**，而不是可有可无的增强层。

## 3. 目标用户与核心场景

### 3.1 目标用户

| 用户类型 | 描述 | 核心痛点 | 使用频率 |
| --- | --- | --- | --- |
| **电商卖家** | Amazon / Shopify / 独立站店主 | 不清楚不同场景下该如何处理图片 | 每周 2-3 次 |
| **内容运营** | 博客、落地页、SEO 内容运营 | 需要兼顾图片质量、体积和元数据 | 每周 1-2 次 |
| **代运营团队** | 批量处理多个客户素材 | 需要快速生成稳定方案并批量执行 | 每周 2-4 次 |
| **不会配参数的普通用户** | 只知道想要结果，不懂图片处理细节 | 面对参数面板会犹豫和流失 | 不固定 |

### 3.2 典型场景

#### 场景 A：Shopify 商品图准备

- 用户输入：“把这批图处理成适合 Shopify 商品页的版本”
- Copilot 输出计划：统一尺寸、压缩、转格式、建议文件名、可选 alt text

#### 场景 B：博客配图优化

- 用户输入：“把这些博客配图压到 500KB 内，优先保证清晰度”
- Copilot 输出计划：WebP 转换、目标压缩区间、尺寸保留策略、命名规则

#### 场景 C：Amazon 图片准备

- 用户输入：“帮我准备 Amazon 主图和辅图”
- Copilot 输出计划：按模板区分主图/辅图规则、给出处理步骤和风险提示

#### 场景 D：多语言内容站图片包

- 用户输入：“导出一份适合英文站使用的图片包，并生成基础 SEO 元数据”
- Copilot 输出计划：格式转换、体积优化、文件名生成、alt text 生成

## 4. 核心产品形态

### 4.1 不做 Chat，做单轮规划式 Copilot

MVP 不采用多轮聊天窗口，而采用以下交互：

- 上传图片
- 在单个输入框中描述目标
- AI 一次性生成结构化任务计划
- 用户确认后执行

该形态更像：

- Command Bar
- AI Task Planner
- Copilot Panel

而不是长对话机器人。

### 4.2 为什么不做 Chat

- 多轮聊天会显著增加 token 消耗
- 长对话不适合高频批量图片处理场景
- 用户真正需要的是“给出目标 -> 得到计划 -> 立即执行”
- 单轮规划更利于控制成本、提高可解释性和前端执行稳定性

### 4.3 页面核心模块

页面应包含以下 7 个核心区域：

1. 页面头部说明区
2. 账号状态入口区
3. 图片上传区
4. AI 目标输入区
5. 计划预览区
6. 执行进度区
7. 导出结果区

## 5. 用户流程

### 5.1 主流程

```text
访问 Workflow Copilot 页面
  -> 上传图片
  -> 输入处理目标
  -> 点击“生成 AI 计划”
  -> 若未登录，则弹出登录
  -> 登录成功后检查免费额度
  -> AI 生成结构化计划
  -> 用户查看并确认计划
  -> 本地执行处理
  -> 导出结果包
```

### 5.2 状态机

| 状态 | 定义 | 主操作 |
| --- | --- | --- |
| `empty` | 未上传图片、未输入目标 | 上传图片 |
| `draft` | 已上传图片，待输入目标 | 输入目标 |
| `pending_auth` | 用户准备生成 AI 计划，但尚未登录 | 登录 |
| `planning` | AI 正在生成计划 | 查看规划进度 |
| `plan_ready` | 计划已生成，待确认 | 审核计划 / 确认执行 |
| `executing` | 本地执行处理中 | 查看进度 |
| `complete` | 处理完成 | 下载结果 |
| `quota_locked` | 免费额度已用完 | 去订阅 |
| `error` | 规划或执行失败 | 重试 |

### 5.3 未登录与已登录的行为差异

#### 未登录用户

- 可访问页面
- 可上传图片
- 可输入目标
- 点击“生成 AI 计划”时必须先登录
- 不可直接调用 AI 规划

#### 已登录免费用户

- 可使用 **3 次免费 AI 规划机会**
- 每次机会包含一次完整的：
  - AI 计划生成
  - 计划确认
  - 本地执行
  - 结果导出

#### 已订阅用户

- 可继续使用 Copilot
- 解锁更高批量上限与持续使用能力

## 6. 登录、免费额度与订阅规则

### 6.1 登录策略

AI 规划必须登录后才能使用。

原因：

- 需要绑定免费额度
- 需要控制 AI 调用成本
- 需要建立订阅路径

MVP 推荐仅支持：

- Google 登录

### 6.2 免费额度定义

#### 免费规则

- 每个注册用户拥有 **3 次免费完整体验**
- 这 3 次为**一次性试用额度**，不是每日刷新
- 每次成功生成 AI 计划即消耗 1 次额度

#### 一次免费体验的定义

一次免费体验包括：

- 1 次 AI 计划生成
- 对当前批次图片的 1 次执行机会
- 1 次结果导出

#### 不消耗额度的情况

以下情况不扣减免费次数：

- 用户上传图片但未点击“生成 AI 计划”
- 登录失败
- AI 服务端超时且未返回有效计划
- 前端在用户确认执行前主动取消

### 6.3 免费额度耗尽后的行为

当用户 3 次额度用完后：

- “生成 AI 计划”按钮置为订阅引导态
- 页面保留上传和输入框，但不可继续调用 AI 规划
- 弹出付费说明面板

### 6.4 订阅规则

#### MVP 订阅策略

- 提供单一 Pro 订阅档位
- 订阅后可继续使用 Workflow Copilot
- 订阅包含合理使用上限，防止滥用

#### Pro 权益

- 解锁 AI 规划继续使用
- 提升单批图片上限
- 解锁更多任务模板
- 解锁历史计划记录
- 解锁增强版导出选项

#### 推荐默认限制

| 用户类型 | AI 规划次数 | 单批图片上限 | 历史记录 |
| --- | --- | --- | --- |
| 未登录 | 0 | 可上传但不可规划 | 无 |
| 免费登录用户 | 3 次总试用 | 20 张 | 无 |
| Pro 订阅用户 | 持续可用（合理使用） | 100 张 | 有 |

### 6.5 账号入口与个人中心边界

当前阶段不做完整个人中心页面，只做一个轻量账号状态入口。

该入口必须能让用户快速看到：

- 当前登录状态
- 当前计划类型：`Free / Pro`
- 剩余免费次数：如 `2 / 3`
- 升级入口
- 退出登录

推荐形式：

- 顶部头像菜单
- 或右上角轻量抽屉

当前阶段明确不做：

- 独立的 Profile 页面
- 账单中心
- 历史支付记录页
- 团队管理页
- 复杂偏好设置页

## 7. 功能需求

### 7.1 图片上传

#### 功能目标

允许用户上传单张或批量图片，作为规划上下文输入。

#### 需求说明

- 支持拖拽上传与点击上传
- 支持多图批量导入
- 支持已有格式检测逻辑
- 前端生成基础特征摘要
- 生成缩略图与批次概览

#### 不直接上传原图给 AI

AI 规划阶段优先只接收：

- 文件数量
- 文件名
- 宽高
- 格式
- 大小
- 基础元数据
- 当前批次统计摘要

### 7.2 AI 目标输入框

#### 功能目标

让用户用自然语言表达处理目标，而不是手工配置参数。

#### 需求说明

- 单输入框，不做聊天历史
- 支持预设提示词
- 支持推荐快捷任务按钮
- 输入框内容必须与当前批次一起提交给规划接口

#### 推荐预设按钮

- `For Shopify`
- `For Amazon`
- `Compress for blog`
- `Prepare for SEO`
- `Convert to WebP`

### 7.3 AI 计划生成

#### 功能目标

根据“用户目标 + 图片批次摘要 + 模板知识”，生成结构化处理计划。

#### 输出必须结构化

AI 输出不能是纯文本段落，必须是结构化 plan。

推荐输出结构：

```json
{
  "taskSummary": "Prepare images for Shopify product pages",
  "confidence": 0.91,
  "steps": [
    {
      "action": "resize",
      "enabled": true,
      "params": {
        "width": 2048,
        "height": 2048,
        "mode": "fit"
      },
      "reason": "Shopify product pages benefit from consistent large square images"
    },
    {
      "action": "convert_format",
      "enabled": true,
      "params": {
        "targetFormat": "webp"
      },
      "reason": "Reduce file size while keeping quality"
    },
    {
      "action": "compress",
      "enabled": true,
      "params": {
        "quality": 82,
        "targetMaxKB": 500
      },
      "reason": "Improve loading speed for storefront pages"
    },
    {
      "action": "generate_alt_text",
      "enabled": true,
      "params": {
        "language": "en"
      },
      "reason": "Improve SEO and accessibility"
    }
  ],
  "risks": [
    {
      "type": "possible_quality_loss",
      "message": "Some large PNG files may lose fine detail after conversion"
    }
  ],
  "exportPlan": {
    "zipName": "shopify-product-images",
    "filenamePattern": "{slug}-{index}",
    "includeMetadataCsv": true
  }
}
```

### 7.4 计划预览与确认

#### 功能目标

在真正执行前，让用户看到 AI 计划，并拥有最终确认权。

#### 需求说明

- 按步骤展示每个动作
- 展示每个动作的理由
- 展示潜在风险
- 允许关闭部分步骤
- 允许调整少量高价值参数
- 点击确认后开始执行

#### MVP 可调整项

- 输出格式
- 压缩质量
- 是否生成 alt text
- 是否重命名
- 是否导出 CSV 元数据

### 7.5 本地执行层

#### 功能目标

将已确认的结构化 plan 映射为现有图片处理能力的调用序列。

#### 执行原则

- 优先在浏览器本地执行
- 复用现有压缩、转换、尺寸调整、ZIP 下载能力
- AI 不直接处理二进制文件
- AI 只负责计划，不负责执行

### 7.6 结果导出

#### 功能目标

让用户获得处理后的图片包和附加导出物。

#### MVP 导出项

- 图片 ZIP
- 处理摘要
- 可选 CSV 元数据

## 8. LangChain 与 AI 角色定义

### 8.1 LangChain 的核心职责

LangChain 在本产品中不负责直接处理图片，而负责：

- 理解用户目标
- 调用 RAG 检索场景知识
- 生成结构化处理计划
- 对计划给出原因和风险说明
- 输出可执行 JSON

### 8.2 LangChain 在流程中的阶段

#### 阶段 1：任务理解

输入：

- 用户自然语言目标
- 当前批次摘要

输出：

- 任务类型
- 目标场景
- 优先级

#### 阶段 2：知识检索

输入：

- 任务类型
- 模板场景

输出：

- Shopify / Amazon / SEO / 博客配图等知识上下文

#### 阶段 3：结构化计划生成

输出：

- 动作列表
- 参数建议
- 风险说明
- 导出方案

#### 阶段 4：解释与展示

输出：

- 面向用户的理由说明
- 关键风险说明
- 为什么建议这样做

### 8.3 LangChain 不负责的事情

- 上传图片
- 提取原始二进制特征
- 实际压缩 / 转换 / 调整尺寸
- ZIP 导出

### 8.4 AI 不可替代性的来源

本产品中 AI 的不可替代性来自于：

- 用户输入的是目标，不是参数
- 系统需要把自然语言目标转成结构化执行计划
- 系统需要结合不同场景知识做合理决策

这使得 AI 成为**前置决策入口**，而不是后置描述层。

## 11. 风险与应对

| 风险 | 说明 | 应对 |
| --- | --- | --- |
| AI 计划不稳定 | 相同目标可能输出不一致 | 固定输出 schema + 模板知识检索 + plan 校验 |
| 计划不可执行 | AI 生成了超出能力边界的动作 | 前端和服务端双层动作校验，只允许白名单 action |
| token 成本过高 | 长对话会带来高成本 | 不做 chat，只做单轮规划 |
| 登录过早导致流失 | 先登录会损失大量用户 | 允许先上传和输入目标，生成计划前再要求登录 |
| 免费额度滥用 | 3 次免费可能被薅 | 登录绑定账号，成功计划才扣额度 |

- 登录 + 3 次免费额度
- 轻量账号状态入口

- 长对话上下文记忆

- 完整个人中心页面

- 高级视觉生成能力

### 13.3 最终结论

`Workflow Copilot` 是 PixelSwift 从“参数驱动的图片工具”升级为“目标驱动的图片工作流产品”的关键一步。

它解决的不是“如何再加一个 AI 功能”，而是：

> **如何让用户只描述目标，就能获得一套可信、可解释、可执行的图片处理方案。**

这使得 AI 在产品中承担真正的前置决策角色，也让登录、免费试用和订阅转化路径具备明确的产品价值支点。
