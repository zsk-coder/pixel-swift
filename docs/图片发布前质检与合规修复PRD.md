## 1. 背景与目标

### 1.1 背景

PixelSwift 当前已经具备图片压缩、格式转换、尺寸调整等底层能力，但这些能力仍停留在“单点处理工具”层面。真实业务中，图片在发布前还存在大量高频且容易返工的问题，包括：

- 图片尺寸、比例、格式、体积不符合发布场景要求
- 商品图背景不干净、主体偏移、清晰度不足
- 文件命名混乱，无法直接交付或上传
- alt text 缺失或质量过低，影响 SEO 与无障碍
- 用户处理完图片后仍需人工做一轮检查和整理

对于 Shopify / 独立站运营与代运营团队来说，真正的痛点不是“不会压缩图片”，而是“图片处理完后，如何在发布前快速确认这批图是否可用，并将能修的问题自动修复掉”。

### 1.2 产品目标

打造一个面向发布前场景的图片质检与合规修复工作流，帮助用户在一次流程内完成：

`上传 -> 选择模板 -> 扫描 -> 问题分级 -> 一键修复 -> 人工复核 -> 导出`

第一阶段不做完整电商上架工作流，不做复杂工作流编辑器，只聚焦“发布前最后一公里”的质量门禁与自动修复。

### 1.3 目标结果

- 用户可以在单一页面中完成整批图片的发布前检查
- 系统可以自动识别阻断级、警告级、建议级问题
- 用户可以对可自动修复的问题执行一键修复
- 用户可以获得可交付的结果包与报告
- 产品具备明确的免费边界、付费升级点与转化路径

### 1.4 核心成功指标

- 首次使用用户可在 3 分钟内完成首批图片扫描
- 首批次扫描完成率 >= 70%
- 首批次用户点击“一键修复”比例 >= 35%
- 导出报告或导出结果包转化率 >= 20%
- 经用户反馈确认，可减少至少 30% 的人工检查时间

## 2. 用户画像与高频场景

### 2.1 主用户画像：Shopify / 独立站运营

| 维度 | 描述 |
| --- | --- |
| 角色 | 店铺运营、商品运营、内容运营 |
| 工作目标 | 快速上新、少返工、避免商品图不合规、提升页面加载与基础 SEO 表现 |
| 常见痛点 | 一批商品图规格不统一、上传前仍需人工检查、平台/场景要求记不住 |
| 购买动机 | 节省发布前整理与审图时间，减少错误和返工 |
| 典型设备 | 桌面端为主，偶尔移动端查看结果 |

### 2.2 次用户画像：代运营团队 / 外包团队

| 维度 | 描述 |
| --- | --- |
| 角色 | 代运营、外包设计交付、站点优化服务商 |
| 工作目标 | 同时服务多个客户，保证交付一致性与可复用性 |
| 常见痛点 | 不同客户图片要求不同、批量交付时易漏图错图、需要留痕和报告 |
| 购买动机 | 提高批量处理效率，沉淀模板，降低客户返工率 |
| 典型设备 | 桌面端，批处理与导出占比高 |

### 2.3 主用户高频场景

#### 场景 A：商品图发布前检查

- 触发时机：运营完成商品图基础处理后，准备上传到 Shopify / 独立站
- 输入：10-50 张商品图
- 目标：确认图片可发布，能修的自动修，最后导出可用图片包
- 成功标准：阻断级问题被发现，可修项被自动处理，用户无需手工逐张复查全部细节

#### 场景 B：落地页 / 博客配图发布前检查

- 触发时机：内容页准备上线前
- 输入：3-20 张页面配图
- 目标：检查体积、尺寸、格式、文件命名和 alt text 是否合适
- 成功标准：页面图符合发布要求，减少性能问题与后续补救工作

### 2.4 次用户高频场景

#### 场景 C：客户批量交付前质检

- 触发时机：代运营团队完成一批客户图片处理后
- 输入：20-100 张图
- 目标：统一执行模板化检查，输出报告和修复结果
- 成功标准：可直接对客户说明问题与修复结果，降低返工与沟通成本

#### 场景 D：模板复用式多批次处理

- 触发时机：团队处理多个客户或多个店铺的相似任务
- 输入：多批次图片
- 目标：通过固定模板快速复制同类检查与修复策略
- 成功标准：重复流程无需重新配置，团队交付一致性提升

## 3. 问题定义与价值主张

### 3.1 当前问题

用户目前通常需要在多个工具之间切换完成：

- 压缩
- 调整尺寸
- 转换格式
- 检查命名
- 生成 alt text
- 打包导出

这些步骤在“发布前”场景下高度耦合，但市场上大多数工具只覆盖其中一个点，导致用户仍需人工兜底。

### 3.2 本产品解决的问题

本产品解决的是**发布前最后一轮质量门禁与自动修复问题**，而不是通用修图问题。

具体表现为：

- 帮用户快速判断“这批图现在能不能发”
- 将问题按优先级分层，而不是给出散乱建议
- 对可自动修复的问题提供一键修复
- 对 AI 不确定的问题提供人工复核入口
- 对结果提供可交付报告和导出物

### 3.3 核心价值主张

**一句话价值主张**：

> 在图片发布前，帮用户一次性完成批量检查、自动修复与结果交付。

### 3.4 与现有 PixelSwift 能力的关系

- 压缩、改尺寸、转格式等仍由 PixelSwift 底层处理能力承接
- 新功能作为一个新的工作流入口，负责“检查 + 决策 + 修复 + 导出”
- 本功能是从“工具”升级到“工作流”的关键节点

## 4. 产品范围、边界与非目标

### 4.1 Phase 1 产品范围

第一版必须实现以下范围：

- 单页面发布前质检工作流
- 2-3 个固定模板
- 批量上传与批次处理
- 规则引擎质检
- AI 辅助判断与文案生成
- 一键修复
- 人工复核
- 结果导出
- 付费边界与升级提示

### 4.2 第一版固定模板

第一版模板固定为：

1. Shopify 商品图
2. 独立站落地页 / 博客配图
3. 通用社媒封面图

说明：

- 模板在第一版中是可执行规则集，不是视觉标签
- 每个模板都必须绑定规则集、阻断项、自动修复项、导出物定义

### 4.3 非目标

第一版明确不做：

- 通用修图工具大全
- 长链路工作流编辑器
- 账号体系 / 团队管理后台
- 通用 DAM / 素材管理系统
- 自动发布到 Shopify / CMS / 社媒平台
- 复杂外部系统打通
- 多平台模板市场
- 高级图像生成、抠图、重绘等重修图能力

### 4.4 Phase 2 演进方向

在第一版验证有效后，Phase 2 才可演进为“商品图上架工作流”，增加：

- 上架模板包
- 平台导出包
- 批量命名与说明生成
- 历史批次复用
- 更强的模板化与外部系统联动

## 5. 核心业务流程与状态机

### 5.1 核心业务流程

```text
上传图片
  -> 选择模板
  -> 开始扫描
  -> 规则引擎检测
  -> AI 分析补充
  -> 结果分级
  -> 一键修复可修问题
  -> 人工复核不确定项
  -> 导出图片包 / 报告
```

### 5.2 页面状态机

| 状态 | 定义 | 主 CTA | 是否允许导出 |
| --- | --- | --- | --- |
| `empty` | 无文件、未开始 | 上传图片 | 否 |
| `uploading` | 正在接收文件 | 上传中 | 否 |
| `scanning` | 规则扫描中 | 扫描中 | 否 |
| `analyzing` | AI 分析中 | 分析中 | 否 |
| `fixing` | 自动修复中 | 修复中 | 否 |
| `ready` | 已出结果，可操作 | 一键修复 / 导出 | 视阻断项而定 |
| `partial_failed` | 部分文件失败 | 查看失败项 / 重试 | 有条件允许 |
| `complete` | 修复与导出已完成 | 再次处理 / 下载结果 | 是 |
| `error` | 整批失败或关键异常 | 重试 | 否 |

### 5.3 阻断逻辑

- 存在阻断级问题且尚未处理时，主导出按钮置灰
- 导出按钮置灰时，必须明确提示阻断原因与受影响文件数
- 若整批中仅部分文件存在阻断问题，允许导出“通过项 + 警告项”，但必须显式二次确认

### 5.4 人工复核触发条件

以下情况进入人工复核：

- AI 置信度低于阈值
- 规则与 AI 判断冲突
- 自动修复可能导致内容裁切或质量损失
- 用户手动要求复核某张图片

## 6. 页面结构与交互规格

### 6.1 页面形态

- 形态：单主页面工作流
- 推荐路径：`/image-audit`
- 设计原则：不像后台台账，而像面向运营与交付场景的专业工具页

### 6.2 页面级结构

页面区块固定顺序如下：

1. 页面头部说明区
2. 上传区
3. 模板选择区
4. 批次概览区
5. 结果列表区
6. 单图详情 / 修复面板
7. 导出区

### 6.3 区块说明

#### 6.3.1 页面头部说明区

- 标题：图片发布前质检与合规修复
- 副标题：批量检查图片问题，一键修复可修项，导出结果与报告
- 说明文案：突出“发布前”“批量”“自动修复”“可导出”

#### 6.3.2 上传区

必须包含：

- 拖拽上传区域
- 点击上传按钮
- 支持多文件批量导入
- 文件数量与大小限制说明
- 支持格式说明

#### 6.3.3 模板选择区

必须包含：

- 模板卡片或模板下拉
- 模板摘要说明
- 当前模板适用场景
- 可选规则开关

#### 6.3.4 批次概览区

必须展示：

- 批次数量
- 通过数
- 警告数
- 阻断数
- 可自动修复数
- 已修复数

#### 6.3.5 结果列表区

列表字段固定为：

- 缩略图
- 文件名
- 模板
- 问题等级
- 问题数量
- 建议动作
- 修复状态
- 导出状态

必须支持：

- 按等级筛选
- 按问题类型筛选
- 只看阻断项
- 只看待修复项

#### 6.3.6 单图详情 / 修复面板

建议采用右侧抽屉或侧栏，不单独跳页。

必须展示：

- 原图预览
- 问题列表
- 每个问题的说明
- 是否可自动修复
- 修复建议
- alt text 建议
- 文件名建议
- AI 置信度
- 手动确认按钮

#### 6.3.7 导出区

必须包含：

- 导出修复后图片 ZIP
- 导出 CSV 报告
- 导出 PDF 报告
- 阻断级问题提示
- 升级提示与付费权益说明

### 6.4 组件级定义

页面内的关键组件固定为：

- 上传器
- 模板选择器
- 规则开关组
- 批次统计卡
- 问题分布图或分布摘要
- 结果表格
- 批量操作栏
- 单图详情抽屉
- 导出按钮组
- 升级提示组件

### 6.5 按钮可用性规则

| 按钮 | 触发条件 | 禁用条件 |
| --- | --- | --- |
| 开始扫描 | 已上传至少 1 张图且已选模板 | 未选模板或无文件 |
| 一键修复 | 扫描完成且存在可修问题 | 扫描中、分析中、无可修项 |
| 导出结果包 | 无未处理阻断项 | 存在阻断项 |
| 导出报告 | 扫描完成 | 扫描失败或无结果 |
| 人工确认通过 | 打开单图详情 | 无 |

### 6.6 桌面端与移动端差异

#### 桌面端

- 左侧为结果列表，右侧为详情 / 修复面板
- 批量操作栏固定在列表顶部
- 导出区固定在页面底部

#### 移动端

- 区块按顺序垂直堆叠
- 详情采用底部抽屉或独立展开卡片
- 批量操作栏吸底
- 列表字段缩减为：缩略图、文件名、等级、动作

### 6.7 文案与术语表

| 术语 | 定义 |
| --- | --- |
| 质检 | 对图片进行规则与 AI 联合检查 |
| 合规 | 满足模板场景要求，可进入发布或交付 |
| 阻断 | 不处理则不能直接导出或发布 |
| 警告 | 可导出，但建议处理 |
| 建议 | 非强制优化项 |
| 自动修复 | 系统可直接执行的修复动作 |
| 人工复核 | 用户手动确认 AI 不确定或高风险结果 |
| 通过 | 图片满足当前模板要求 |

## 7. 功能需求清单

### 7.1 批量上传

#### 功能目标

支持用户一次导入一批图片，作为一个批次进入后续质检流程。

#### 需求说明

- 支持拖拽上传、点击上传
- 支持多文件批量导入
- 限制单图大小与单批数量
- 进入批次后生成唯一批次 ID
- 生成缩略图供结果区展示

### 7.2 模板选择

#### 功能目标

让用户在扫描前明确当前图片的发布场景，避免规则无上下文。

#### 需求说明

- 用户必须在开始扫描前选择模板
- 每个模板包含场景说明、规则集、导出物定义
- 模板选择后才允许扫描

### 7.3 规则质检

#### 功能目标

通过硬规则快速判断图片是否满足基础发布要求。

#### 检查维度

- 尺寸
- 长宽比
- 文件大小
- 文件格式
- 分辨率
- 文件命名规范
- alt text 是否存在

### 7.4 AI 分析

#### 功能目标

补充规则引擎难以稳定处理的软判断与文案生成能力。

#### AI 负责范围

- 背景干净度判断
- 主体偏移判断
- 场景适配判断
- alt text 生成
- 文件名建议生成

### 7.5 一键修复

#### 功能目标

对可自动修复的问题统一处理，减少用户手工操作。

#### 自动修复动作

- 压缩文件体积
- 转换输出格式
- 统一尺寸
- 统一比例
- 批量重命名
- 批量生成 alt text

### 7.6 人工复核

#### 功能目标

为 AI 不确定或风险较高的结果提供最终确认机制。

#### 需求说明

- 支持逐张查看
- 支持保留原图
- 支持采用修复版
- 支持跳过本张
- 支持查看 AI 原始建议与置信度

### 7.7 导出

#### 功能目标

让用户可以直接拿走修复后的结果与说明性报告。

#### 导出形式

- 修复后图片 ZIP
- CSV 报告
- PDF 报告

## 8. 规则引擎、AI 能力与人工兜底

### 8.1 分工原则

采用：

> 规则引擎做硬判断，AI 做软判断，人工做最终兜底

### 8.2 问题等级定义

| 等级 | 定义 | 是否阻断导出 | 默认处理策略 |
| --- | --- | --- | --- |
| 阻断 | 不处理将直接影响发布、交付或模板合规性 | 是 | 必须修复或人工确认跳过 |
| 警告 | 可以导出，但存在明显优化风险或潜在返工风险 | 否 | 默认推荐修复 |
| 建议 | 非强制问题，不影响导出 | 否 | 仅展示建议 |

### 8.3 规则引擎负责内容

- 尺寸是否合法
- 比例是否合法
- 文件体积是否超阈值
- 文件格式是否符合模板
- 文件命名是否缺失或明显混乱
- 是否存在 alt text 缺失

### 8.4 基础规则矩阵

以下基础规则不依赖 AI，必须在规则引擎中完成。

| 规则类型 | 判定条件 | 默认等级 | 是否阻断 | 是否可自动修复 | 失败提示示例 |
| --- | --- | --- | --- | --- | --- |
| 文件数量 | 单批超过 100 张 | 阻断 | 是 | 否 | 当前批次超过上限，请拆分后再处理 |
| 文件大小 | 单张超过 20MB | 阻断 | 是 | 否 | 文件过大，当前版本不支持扫描 |
| 格式合法性 | 非 JPG / PNG / WebP / AVIF 输入 | 阻断 | 是 | 否 | 当前模板暂不支持该格式 |
| 体积过大 | 单张大于模板推荐体积但未超扫描上限 | 警告 | 否 | 是 | 文件体积偏大，建议压缩后导出 |
| 尺寸过小 | 短边小于模板最小阈值 | 阻断 | 是 | 否 | 图片尺寸过小，可能无法满足发布要求 |
| 尺寸不匹配 | 与模板推荐尺寸不一致但可缩放到目标尺寸 | 警告 | 否 | 是 | 尺寸不匹配，建议统一到模板尺寸 |
| 长宽比不匹配 | 与模板目标比例偏差超过 3% | 警告 | 否 | 是 | 当前比例与模板不一致，建议裁切或补边 |
| 文件命名为空 | 文件名为空、纯随机串或含大量无意义字符 | 警告 | 否 | 是 | 文件名不可读，建议重命名 |
| alt text 缺失 | 当前批次未提供 alt text | 警告 | 否 | 是 | 当前图片缺少 alt text 建议 |
| alt text 过短 | alt text 少于 5 个有效词 | 建议 | 否 | 是 | alt text 信息过少，建议补充描述 |

### 8.6 AI 负责内容

- 背景是否足够干净
- 主体是否明显偏移
- 当前图片是否适合目标场景
- 生成 alt text
- 生成推荐文件名

### 8.7 AI 输出约束

AI 必须返回结构化 JSON，不允许只返回自然语言段落。

推荐输出结构：

```json
{
  "summary": "usable_with_warnings",
  "confidence": 0.82,
  "issues": [
    {
      "type": "background_cleanliness",
      "severity": "warning",
      "blocking": false,
      "autoFixable": false,
      "message": "背景存在轻微杂色"
    }
  ],
  "suggestions": {
    "altText": "white ceramic mug on a clean background",
    "filename": "white-ceramic-mug-product-image.webp"
  }
}
```

AI 输出中的 `severity`、`blocking`、`autoFixable` 必须与规则引擎允许的枚举保持一致，不允许新增自由文本等级。

### 8.8 LangChain 约束

- LangChain 作为推荐实现，用于服务端 AI 编排层
- PRD 不将产品能力绑定到 LangChain 框架
- 后续技术规格中可将 LangChain 作为默认推荐实现

### 8.9 自动修复边界

第一版允许自动修复的范围固定为：

- 压缩文件体积
- 转换图片格式
- 统一尺寸到模板目标尺寸
- 对轻微比例偏差执行补边
- 批量生成推荐文件名
- 批量生成 alt text

第一版明确不自动执行：

- 高风险主体裁切
- 自动抠图
- 自动重绘背景
- 对模糊图片做“增强清晰”承诺
- 任何可能改变主体内容语义的 AI 图像修改

### 8.10 人工兜底原则

- AI 置信度不足时，必须允许用户手动覆盖建议
- 自动修复不允许对高风险裁切类操作默认强执行
- 所有阻断级问题必须可解释

### 8.11 技术与非功能约束

#### 8.11.1 处理边界

- 压缩、改尺寸、转格式、ZIP 打包优先在浏览器端完成
- AI 分析必须走服务端接口，不在前端直接暴露模型调用
- 规则引擎必须可在本地直接运行，不依赖 AI 接口可用性

#### 8.11.2 性能边界

| 项目 | 目标值 |
| --- | --- |
| 首屏可交互时间 | <= 3 秒 |
| 单批最大文件数 | 100 张 |
| 单文件最大扫描大小 | 20MB |
| 批量扫描目标耗时 | 20 张图在 30 秒内完成基础扫描 |
| AI 分析超时 | 单张 8 秒，整批可并发受控 |

#### 8.11.3 任务容错

- 单张失败不影响整批继续
- 同一文件默认最多重试 1 次
- 扫描失败、AI 失败、修复失败必须分别记录原因
- 部分失败批次允许继续进入结果页

#### 8.11.4 数据安全

- 默认不保存原图到长期存储
- 浏览器本地处理完成后应及时释放中间资源
- 若服务端 AI 分析需要上传图片，只上传分析所需文件，不长期保留
- 历史批次在 Phase 1 中默认不提供跨天持久化

#### 8.11.5 国际化与无障碍

- 文案必须支持现有多语言体系扩展
- 所有状态标签、按钮、错误提示需可国际化
- 上传区、结果列表、详情抽屉必须支持键盘可达
- 颜色以外还需使用图标或文案区分问题等级

### 9.5 商业化设计

#### 免费边界

- 每月固定免费批次数或免费图片张数
- 免费版可扫描、可查看基础结果
- 免费版不开放高级导出或批量修复上限较低

#### 付费触发点

- 批量导出 ZIP / PDF
- 更多图片额度
- 更多模板
- 历史记录与批次留痕
- 高级 AI 分析

#### 套餐草案

| 套餐 | 定位 | 建议内容 |
| --- | --- | --- |
| Free | 试用 | 基础扫描、少量额度、限制导出 |
| Pro | 个人/小团队 | 更高额度、批量修复、完整导出 |
| Team | 代运营/服务商 | 模板复用、批次留痕、团队交付增强 |

#### 关键转化动作

- 扫描完成后展示可修复问题数量
- 导出前展示升级收益
- 阻断项处理完成后引导导出
- 免费额度耗尽时触发升级提示

## 10. 风险、异常、版本规划与开放问题

### 10.1 主要风险

| 风险 | 说明 | 应对策略 |
| --- | --- | --- |
| 功能边界膨胀 | 容易滑向通用图片工具大全 | 严格限制模板和非目标 |
| AI 结果不稳定 | 软判断可能存在误差 | 结构化输出 + 人工复核 |
| 阻断逻辑不清 | 会影响导出体验 | 文档中写死等级与导出规则 |
| 双人群失焦 | 运营与代运营诉求不同 | 主用户写穿，次用户补充批量和留痕 |

### 10.2 异常与降级策略

- 单张图片扫描失败不影响整批继续处理
- AI 分析失败时，保留规则引擎结果并提示“AI 建议暂不可用”
- 自动修复失败时，用户可选择保留原图并继续导出其他结果
- 导出失败时，保留当前批次结果并允许重试

### 10.3 Phase 1 与 Phase 2 规划

#### Phase 1：发布前质检 / 合规修复

- 批量扫描
- 问题分级
- 自动修复
- 人工复核
- 导出结果与报告

#### Phase 2：商品图上架工作流

- 增加上架模板包
- 增加多场景导出策略
- 增加更强的模板复用能力
- 增加历史批次复用
- 增加面向交付和上架的工作流编排能力

### 10.4 当前开放问题

以下问题在本 PRD 中已给出推荐方向，但后续技术规格可继续细化：

- AI 置信度阈值的默认值
- 各模板规则的精确阈值
- 免费额度的最终数值
- PDF 报告的最终样式

### 10.5 最终结论

本功能不是“又一个图片工具”，而是 PixelSwift 从单点处理能力升级到工作流能力的第一阶段。  
通过将规则引擎、AI 辅助判断、自动修复与结果导出整合在单主页面中，本功能将成为后续 AI 生成 UI、技术规格设计、开发实现和商业化落地的统一基础文档。
