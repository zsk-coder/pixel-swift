# PixelSwift 验证顺序

## 默认验证顺序
开发新功能、修 Bug、改 UI、改文案时，默认按下面顺序验证：

1. 阅读相关页面、组件、composable、server API、locale 现状
2. 完成实现
3. **先跑 `npm run dev`**
4. 打开真实页面/路由手动验证是否报错、是否符合预期
5. 再跑与本次改动直接相关的测试
6. 最后才跑 `npm run build`

## 为什么不是先 build
- `npm run build` 更慢，更适合做最终回归
- 许多开发期错误更容易在 `npm run dev` + 真实页面访问时第一时间看到
- 尤其是：
  - Nuxt 路由/页面错误
  - composable 自动导入问题
  - SSR/CSR 行为差异
  - locale key 缺失
  - Supabase 环境变量与回调页问题

## 什么时候必须跑 dev
- 新页面
- 新组件接入页面
- 登录/回调/头部/路由相关改动
- i18n 文案改动
- 任何用户真实会访问的交互流程

## 什么时候再补 build
- 功能模块完成后
- 准备交付、合并或声明“已完成”前
- 需要确认生产构建无误时

## 基本原则
- `run dev` 是开发期首选验证
- `run build` 是收尾校验，不是第一反应
